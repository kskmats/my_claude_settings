# Auto Commit

変更内容を分析し、論理的な単位に自動分割してコミットするスキル。

## ワークフロー

### 1. 状態の確認

```bash
git status
git diff HEAD
git log --oneline -5
```

### 2. .gitignoreの確認と更新

未追跡ファイルを確認し、コミットすべきでないファイルが含まれている場合は`.gitignore`に追加する。

対象例:
- `terraform.tfstate`, `.terraform/`
- `node_modules/`, `vendor/`
- `.env`, `.env.local`
- `*.pyc`, `__pycache__/`
- `.DS_Store`

`.gitignore`の更新が必要な場合は、関連する変更と同じコミットに含める。

### 3. 変更の分析と分割判断

以下の基準で変更を論理的な単位に分割する:

- **ファイルの種類・目的が異なるもの**を分ける（例: ソースコードと設定ファイル）
- **機能的に独立した変更**を分ける（例: バグ修正と新機能追加）
- `.gitignore`の更新は関連する変更と同じコミットに含める
- 分割不要な場合（関連する変更のみ）は1つのコミットにまとめる

### 4. コミット実行

分割した各単位について:

1. 対象ファイルを `git add` でステージング（ファイル名を個別に指定、`git add .` は使わない）
2. コミットメッセージを生成
3. `git commit` を実行

## コミットメッセージ形式

```
<type>: <日本語の説明>
```

### typeの選択基準

| type | 用途 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `refactor` | 機能変更を伴わないコード改善 |
| `chore` | ビルド、設定、依存関係などの雑務 |
| `style` | フォーマット、セミコロン等の修正（動作に影響なし） |
| `test` | テストの追加・修正 |

### ルール

- 説明は日本語で記述する
- 1行で完結させる（本文なし）
- スコープは使わない（`feat(auth):` のような形式は不使用）
- Co-Authorshipフッターは不要
- 「何をしたか」を簡潔に記述する

### 例

```
feat: ユーザー認証機能を追加
fix: ログイン時のリダイレクトエラーを修正
chore: .gitignoreにnode_modulesを追加
refactor: API呼び出しを共通関数に統合
docs: READMEにセットアップ手順を追記
```

## 注意事項

- ユーザーに確認を求めず、自動でメッセージ生成・コミットを実行する
- `git push` はコミットのみのスキルなので実行しない
- センシティブなファイル（.env、credentials等）はコミットしない
